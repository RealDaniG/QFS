Session Management System

Authentication Flow

  1. 1
    Device Registration

    Deterministic device ID computed from os_type, hardware_id, and optional label.

  2. 2
    Challenge Request

    Client requests challenge for wallet_id and device_id.

  3. 3
    Challenge Response

    Client signs challenge with wallet private key and submits signature.

  4. 4
    Session Establishment

    Server verifies signature and creates deterministic session.

Session Lifecycle

  • Create Session

    Deterministic session ID generation using SHA-256 from wallet_id, device_id, and timestamp.

  • Rotate Session

    Generate new session with same wallet/device but updated timestamp.

  • Revoke Session

    Mark session as inactive with ledger event emission.

Ledger Events

SESSION_STARTED

Emitted when new session is created with wallet_id, device_id, session_id, and timestamp.

SESSION_ROTATED

Emitted when session is rotated with old_session_id, new_session_id, and timestamp.

SESSION_REVOKED

Emitted when session is revoked with session_id and revocation timestamp.

Explain-This Architecture

Bridging the gap between deterministic QFS economics and user-facing clarity.

Request Flow

  1. 1
    User Action

    User clicks "Explain Reward" in ATLAS Wallet.

  2. 2
    API Request

    GET /explain/reward?tx_id=...

  3. 3
    Value-Node Replay

    Engine loads state at Tx block height, replays logic deterministically.

  4. 4
    Explanation Generation

    Returns factors: Coherence Score, Treasury Cap, Network Load.
